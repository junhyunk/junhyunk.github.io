- 테스트의 목적
    - 어플리케이션에 대한 전반적인 테스트를 통해 이해관계자들의 의사결정에 도움을 줌.
    - 추가적으로, 성능 테스트는 시스템의 병목점을 식별하거나, 튜닝, 향후 테스팅을 위한 사전작업, 그리고 성능 목표와 요구사항을 결정하는 데 사용되는 경향이 있음. 또한, 성능 테스트 결과를 통해 배포 시 필요한 하드웨어 구성을 가늠할 수 있음.


### Chapter 1 을 읽고 성능 테스트의 정의를 정리하시오. (Word 1 page 분량)
- 성능 테스트는 반응성(responsiveness), 처리량, 신뢰성을 결정하거나 주어진 작업량에 대한 시스템의 확장성을 판단하는 데 목적을 둠.
- 성능 테스트는 다음과 같은 것을 이루기 위해 수행됨
    - 제품의 준비도
    - 성능 기준에 대해 실행(?)
    - 다중 시스템이나 설정에 대한 성능 특성을 비교
    - 성능 문제 식별
    - 시스템 튜닝
    - 처리량 식별

- 성능 테스트 핵심 activity
    1. 테스트 환경 식별하기
        - HW, SW, 네트워크 설정 등등
    2. 성능 기준 식별하기
        - response time, throuput, resource utilization goals, constraint 등등
    3. 테스트 계획 및 설계하기
        - 시나리오, 대상 사용자, 테스트 데이터, 수집할 데이터 등등 정의
    4. 테스트 환경 구성하기
        - 테스트 환경, 데이터, 결과값 검증
        - 검증된 테스트 수행해야 됨
    5. 테스트 설계 구현하기
    6. 테스트 실행
    7. 결과 분석, 보고, 테스트 재실행하기
        - 결과 공유..

- Project Context(이하 '프로젝트 맥락')
    - 성능 테스트가 성공적으로 수행되기 위해서는, 성능을 테스트하는 것과 그 자체를 테스트하는 것이(?) 프로젝트 맥락과 관련이 있어야 함(쓸데없는 테스트 하지 말란 소리인 듯)
    - 프로젝트 맥락에 대한 이해가 없다면, 정작 중요한 건 테스트 못해서 시간낭비에, 좌절에, 충돌 등등 안좋음.
    - 프로젝트 맥락은 다음과 같음(아래 항목으로 제한하는 건 아님)
        - 프로젝트의 전체적인 비전(목적)과 의도
        - 성능 테스트의 목적
        - 성능 기준
        - 개발 라이프사이클
        - 프로젝트 일정
        - 프로젝트 예산
        - 가용한 도구와 환경
        - 성능 테스트를 수행하는 팀의 스킬셋
        - 파악된 성능 이슈에서의 우선순위
        - 오동작하는 어플리케이션을 배포했을 경우의 business impact

- 프로젝트의 비전
    - 프로젝트의 비전은 어떤 성능 테스트가 중요하고 가치가 있을지 결정함.
    - 비전이 변경될 가능성이 있다면, 가능한 자주 확인할 것.
- 시스템의 목적
    - 테스트할 어플리케이션이나 시스템의 목적을 이해할 것.
    - 테스트에서 중점을 두어야 하는, 가장 우선순위가 높은 성능 특성을 식별하는 데 도움을 줌.
    - 시스템의 관심사
- 사용자의 목적
    - 성능 테스트를 할 때는 사용자의 기대치를 항상 염두에 둘 것.
    - 사용자의 기대치는 단순히 시스템 요구사항을 준수하는 데 끝나지 않음
- 비즈니스적 요소
    - 예산, 일정, 자원 등에 의해 결정되는 비즈니스적 요구사항과 기회(?)
    - 가용한 예산 내에서 일정 맞추는 데 중요함
- 성능을 테스트하는 이유
    - 성능 테스트를 왜 하는지는 프로젝트 초기 단계에서 이해해야 함.
        - 제대로 이해 안하면 비효율적인 성능 테스트로 이어짐.
        - 성능 기준 초과하고, 자주 바꾸는 등등..
    - 자주 확인할 것
- 성능 테스트가 프로젝트에 주는 가치
    - 프로젝트와 비즈니스 레벨에서의 목적을 구체적이고, 식별 가능하고, 통제 가능한 성능 테스트 활동으로 변환할 것
- 프로젝트 관리
    - 성능 테스트를 효율적으로 수행하기 위해서 팀의 구성, 역할, 커뮤니케이션 기술을 이해할 것
- 프로세스
    - 팀의 개발 프로세스를 이해하고, 프로세스 문서에 성능 테스트 과정이 명시되어 있지 않다면 적절하게 수정해서 PM에게 승인받을 것.
- Compliance criteria(규정 준수?)
- 프로젝트 일정
    - 프로젝트의 시작일과 종료일, 하드웨어의 가용 기간, 빌드와 배포, 체크포인트와 마일스톤에 주의할 것.

- 성능 테스트와 튜닝의 관계
    - 성능 테스트 결과, 해당 특성(특정 테스트 결과인듯)이 기준을 통과하지 못할 경우(의역) 대개 성능 테스트에서 성능 튜닝으로 관점을 옮김.
    - 성능 튜닝은 성능 테스터의 직접적인 책임은 아니지만, 튜닝 프로세스는 다음과 같은 관계자들이 협력할 때 가장 효율적임.
        - 제품 벤더
        - 설계자
        - 개발자
        - 테스터
        - DB 관리자
        - 시스템 관리자
        - 네트워크 관리자
    - 팀간 협력 이루어지지 않으면 성능이슈 해결하기 위한 시스템적인 관점을 얻는 것이 거의 불가능함.
    - 성능 테스터나 팀은 이러한 협력관계에서 핵심적인 역할을 함.
        - 보통 다양한 부하(Load) 상황과 환경에서의 컴포넌트, 자원, 응답에 대한 추가적인 모니터링이 필요함.

- 전반적인 튜닝 과정
    - 성능 튜닝은 성능 테스팅과 보통 분리되어 있지만 독립적이지는 않음.

    1. 테스트는 잘 정의되고 통제된 환경에서 수행됨.
        - 테스트 환경과 결과가 시작 시 정해져 있고, 재생산 가능함.
    2. 테스트 결과 해당 성능 특성이 기준을 통과하지 못할 경우, 성능 테스트와 튜닝 팀은 진단 및 개선 단계로 돌입함.
        - 테스트 환경이나 어플리케이션의 변화를 요구하게 될 것임.
    3. 튜닝 단계에서의 효율성을 극대화하기 위해, 테스트와 튜닝 팀에게는 full-exclusive(admin 같은거)한 테스트 환경이 주어진다. 
    4. 튜닝 프로세스는 빠른 변화와 테스트 시퀀스가 포함된다.
        - 협력 관계에 있는 팀들이 가용하지 않거나 제대로 일 안하면 효율 떨어짐.
    5. 튜닝 단계가 끝나면 테스트 환경은 보통 초기 상태로 되돌림.
        - 반영할거 반영해서 다시 테스트함.

- 성능(Performance), 부하(Load), 스트레스(Stress) 테스트
    1. 성능 테스트
        - 시스템이나 어플리케이션의 속도, 확장성, 또는 안정성을 테스트함.
        - 성능은 프로젝트나 제품이 목표로 하는 응답 시간, 처리량, 자원 사용률(utilization)에 따라 결정됨.
        - 본 책에서는 다른 성능과 관련된 모든 테스트의 superset이라 정함.
        
        1. 부하 테스트
            - 제품이 동작하는 동안, 주어진 작업량에 대한 성능 특성을 검증하는 것.
        2. 스트레스 테스트
            - 제품이 동작하는 동안, 주어진 조건에 대한 성능 특성을 검증하는 것.
            - 제한된 메모리, 부족한 디스크 공간, 서버 실패 등의 조건 등에 대한 테스트가 될 수 있음.
            - 어떤 조건에서 어플레케이션이 fail하는지, 어떻게 fail하는지, fail하는 동안 모니터링되는 지표는 무엇인지 식별함.
        
- 기준치(Baseline)
    - 기준치를 정하는 건 시스템이나 어플리케이션의 성능을 점진적으로 개선하는 데 지표가 됨.
    - 기준치에서 중요한 건 모든 특성이나 환경 중, 비교를 위한 항목을 제외하고는 변하면 안 된다는 것임.
    - 로딩 시간, 단위 시간당 처리한 트랜잭션 개수, 단위 시간당 처리한 웹페이지 개수, 메모리 사용량, 프로세서 사용량 등

- 벤치마킹
    - 내부적으로 정한 기준치나, 다른 단체에서 정의한 산업표준에 대해 시스템의 성능을 비교하는 것.
    - 벤치마킹은 다음과 같은 사항을 고려함
        - 규칙에 의거해야 함
            - 벤치마크는 산업에서 정한 spec을 구현하거나, 이미 표준에 맞도록 구현된 것들을 가져다 쓰는 것으로 실행됨.
        - 규칙에 의거해야 하기 때문에, 투명성을 가져야 함
            - 벤치마크 결과는 외부로 공개될 수 있음.
            - 경쟁자들에 의해서 비교 결과가 산출될 수 있기 때문에, 신뢰할 수 있는 결과를 얻기 위해 엄격한 기준과 방법을 도입할 필요가 있음.
        - 다양한 측정치를 거쳐 결과를 공개함.
            - 단위 시간당 처리한 ... 개수, ... 사용량
        
- 용어

- 요약
    - 성능 테스트는 시스템의 병목점을 식별하고, 향후 테스팅을 위한 사전작업과, 성능 튜닝을 위한 지원, 성능 목표와 요구사항을 결정하는 데 도움을 줌.
    - 개발 라이프사이클 초기에 성능 테스트를 도입한다면 프로젝트에 중요한 가치를 부여할 수 있음.
    - 성공적으로 성능 테스트를 하기 위해서는, 테스트는 프로젝트 맥락과 반드시 관련이 있어야 함.
        - 뭐가 정말로 중요한지 집중할 수 있도록
    - 성능 테스트 결과, 해당 특성(특정 테스트 결과인듯)이 기준을 통과하지 못할 경우(의역) 대개 성능 테스트에서 성능 튜닝으로 관점을 옮김.
        - 향후 시스템 성능을 개선하기 위해서라도, 자원 사용량을 줄이기 위해 튜닝에 집중할 수 있음.
    - 성능, 부하, 스트레스 테스트는 모두 성능 테스트의 subcategory임. 각각 다른 목적이긴 하지만.

### Chapter 2 를 읽고 성능 테스트의 종류를 정리하고, 해당 테스트가 어떤 상황에서 필요한지 직접 생각하여 정리하시오. (Word 2 page 분량)
- 개요
    - 성능 테스트는 성능과 관련된 테스트들을 의미하는 일반적인 용어임.

- 성능 테스트의 핵심 유형
    - 성능 테스트
        - 속도, 확장성, 안정성 등을 검증.
    - 부하 테스트
        - 보통 또는 최고 부하 상태일 때 동작을 검증.
    - 스트레스 테스트
        - 과부하 상태에서 어플리케이션의 버그를 검출.
        - 버그에는 동기화 이슈, race condition, 메모리 누수 등이 있음
    - 용량 테스트
        - 사용자나 트랜잭션이 많아져도 목표한 성능을 만족하는지 검증.

- 웹앱에서 가장 관심을 두는 성능관련 사항은 다음과 같음
    - 충분히 빠른가?
    - 모든 클라이언트를 감당할 수 있는가?
    - 뭔가 잘못되면 어떻게 되는거지?
    - 고객이 많아지면 어떤 계획을 세워야 되지?
- 즉, 성능 테스트로 충분히 빠른 것을 검증하고
- 부하 테스트로 현재/예상 사용자만큼 수용할 수 있는지 검증하고
- 용량 테스트로 향후 성장에 따른 계획을 검증함.

- 성능 테스트의 핵심 유형별 이점과 한계
    - 성능 테스트
        - 속도, 확장성, 안정성 등을 검증.
        - 성능 특성들 충분히 만족하는지, 예상과 실제로 다르지 않는지 확인함.
        - 튜닝, 수용 계획, 최적화 등에 도움됨.
        - 부하 상황에서의 기능적 결함은 발견하지 못할 수 있음.
        - 테스트가 제대로 설계되고 검증되지 않는다면, 성능 특성 중 소수만을 확인할 수도 있음.
    - 부하 테스트
        - 보통 또는 최고 부하 상태일 때 동작을 검증.
        - 하드웨어 환경, 로드밸런서 등이 충분한지 확인.
        - concurrency 이슈, 부하 상황에서의 기능 오류 검출.
        - 확장성이나 수용 계획에 필요한 데이터 수집.
        - 얼마나 많은 사용자까지 감당할 수 있는지 확인.
        - 하드웨어 자원 사용률을 초과하지 않는 한도에서, 얼마나 많은 부하를 감당할 수 있는지 확인.
        - 반응속도를 확인하기 위한 것이 주된 목적은 아님.
        - 테스트 결과는 반드시 연관된 다른 부하 테스트 결과와만 비교되어야 함.
    - 스트레스 테스트
        - 과부하 상태에서 어플리케이션의 버그를 검출.
        - 버그에는 동기화 이슈, race condition, 메모리 누수 등이 있음.
        - 어느 정도까지의 부하 상황에서 사용자의 요청이 실패하거나 오류를 발생하지 않는지 확인.
        - 스트레스 상황에서 보안 취약점은 없는지 확인.
        - 하드웨어나 어플리케이션이 실패한다면 side-effect는 무엇인지 확인.
        - 스트레스 테스트는 비현실적으로 설계되기 때문에, 일부 이해관계자들이 무시할 수도 있음.
        - 얼마나 많은 스트레스가 적용되어야 하는지 정하기 어려움.
        - 테스트 환경이 잘 분리되어 있지 않다면 (실서버에서의) 네트워크 오류 발생할 수 있음.
    - 용량 테스트
        - 사용자나 트랜잭션이 많아져도 목표한 성능을 만족하는지 검증.
        - 테스트 만들기 복잡함.
        - 항상 모든 측면에서 검증할 수는 없음.

- 성능 테스트를 수행함으로써 얻는 잠재적인 이점이 테스트의 한계점보다 훨씬 큼에도, 결과 데이터에 대한 불확실함(모든 타당한 상황, 조건, 시나리오에 대한 테스트가 어렵기 때문) 때문에 의문점이 생길 수 있음.
- 실제로는 성능 테스트를 통해 어플리케이션이 fail하는 지표를 식별할 수 있기 때문에, 시스템이 성능 때문에 완전히 죽는 상황은 많이 줄어듦.

- 추가 개념/용어
    - 컴포넌트 테스트
    - Investigation
    - Smoke test
    - Unit test
    - Validation test

### Chapter 3 을 읽고 성능테스트시에 어떤 위험사항이 있는지 정리하시오. (Word 1 page 분량)

* 성능 테스트 자체의 위험사항이 아니라, 성능 테스트가 다루는 위험사항을 의미함.

- 개요
    - 성능 테스트는 심각한 위협을 관리하는 데 있어 필수적임.
    - 많은 기업 및 성능 테스터들은 성능 테스트가 속도, 확장성 및 안정성 세 가지 범주로 해결할 수 있는 위험을 고려하는 것이 중요하다고 생각함.
    - 속도, 확장성, 안정성이 성능 테스트의 관점에서 어떻게 다뤄지는지 이해하자.

- 성능 테스트 유형별 다루는 위험사항들
    - 용량 테스트
        - 보통 또는 최고 부하 상황일 때 용량이 비즈니스 기준을 만족하는가?
    - 컴포넌트 테스트
        - 해당 컴포넌트가 기대치를 충족하는가?
        - 해당 컴포넌트가 타당하게 최적화되었는가?
        - 발견된 성능 이슈가 해당 컴포넌트 때문인가?
    - 인내(Endurance) 테스트
        - 시간이 지나도 성능이 일정한가?
        - 감지되지는 않았지만 천천히 증가하는 문제가 있는가?
        - 의도하지 않은 외부로부터의 간섭이 있는가?
    - 검사(Investigation) 테스트
        - 시간에 따른 성능 경향을 측정하는 방법은 어떤 것인가?
        - 향후 테스트를 위해 무엇을 비교해야 하는가?
    - 부하 테스트
        - 어플리케이션이 특정 작업량을 목표로 하고 있을 때, 비정상적인 동작을 하기까지 얼마나 많은 사용자를 감당할 수 있는가?
        - DB 서버는 얼마나 많은 데이터를 감당할 수 있는가?
        - 네트워크 컴포넌트는 적절한가? 
    - Smoke 테스트
        - 빌드/설정 등이 추가적인 성능 테스트를 위해 준비되어 있는가?
        - 차후 수행해야 할 성능 테스트는 어떤 유형인가?
        - 마지막 빌드보다 현재 빌드가 성능이 좋은가/나쁜가?
    - Spike 테스트
        - 예상되는 최고 부하보다 큰 부하가 발생한다면 어떤 일이 발생하는가?
        - 어떤 fail 상황에 대해 계획(대비)해야 하는가?
        - 어떤 지표들을 살펴봐야 하는가?
    - 스트레스 테스트
        - 예상되는 부하보다 큰 부하가 발생한다면 어떤 일이 발생하는가?
        - 어떤 fail 상황에 대해 계획(대비)해야 하는가?
        - fail하기 전에 개입하기 위해, 어떤 지표들을 살펴봐야 하는가?
    - 유닛 테스트
        - 코드 세그먼트는 효율적인가?
        - 성능 예산(budget) 범위 내에 있는가?
        - 해당 코드가 예상되는 부하 상황에서 동작하는가?
    - 검증 테스트
        - 어플리케이션이 목표와 요구사항을 만족하는가?
        - 마지막 버전보다 현재 버전이 빠른가/느린가?
        - 배포하게 된다면 계약 조건을 위반하지는 않는가?

- 속도 관련 위험사항
    - 속도 관련 위험사항은 대부분 사람들이 생각하듯 최종 사용자단의 만족에만 국한되지 않음.
    - 속도는 특정한 비즈니스, 데이터 관련 위협에 대한 지표이기도 함.
    - 성능 테스트가 확인할 수 있는 일반적인 속도 관련 위험사항은 다음과 같음.
        - 어플리케이션이 최종 사용자를 만족시킬 수 있을 만큼 충분히 빠른가?
        - 어플리케이션이 수집한 데이터가 기한이 지나기 전에 활용할 수 있는가?
        - 어플리케이션은 최신의 정보를 사용자에게 제공할 수 있는가?
        - 웹 서비스는 에러가 발생하기 전에 최대 예상 응답 시간 안에 응답하는가?

- 확장성 관련 위험사항
    - 확장성 관련 위험사항은 어플리케이션이 지원할 수 있는 사용자의 수 뿐만 아니라, 보유하고 있는 용량에 대해 보유하고 관리할 수 있는 데이터의 크기에 관련함.
    - 성능 테스트가 확인할 수 있는 일반적인 확장성 관련 위험사항은 다음과 같음.
        - 어플리케이션이 전체 사용자에 대해 일정하고 수용할 수 있는 응답 시간을 제공하는가?
        - 어플리케이션이 수행되는 동안 수집된 모든 데이터를 저장할 수 있는가?
        - 어플리케이션이 용량의 한계에 도달한다는 지표가 있는가?
        - 많은 사용량 중에도 보안을 유지하는가?
        - 많은 사용량 중에 기능이 제대로 동작하지 않는가?
        - 최고 부하 상태에서도 견딜 수 있는가?

- 안정성 관련 위험사항
    - 안정성은 신뢰성, 가동, 복구성을 포함함.
    - 안정성 위험사항은 일반적으로 높은 부하, 내구성, 스트레스 테스트를 통해 나타남.
    - 안정성 이슈는 가장 기본적인 성능 테스트에서 나타나기도 함.
    - 성능 테스트가 확인할 수 있는 일반적인 안정성 관련 위험사항은 다음과 같음.
         - 어플리케이션이 오랜 기간동안 데이터 손상, 속도 저하, 서버 재부팅 없이 동작할 수 있는가?
         - 어플리케이션이 예기치 않게 중단되었을 경우, 일부만 완료된 트랜잭션에 대해서는 어떤 일이 발생하는가?
         - 어플리케이션이 중단되었다가(예정되었건 아니건) 다시 동작하는 경우, 사용자들은 기대했던 모든 것(동작)을 할 수 있는가?
         - 예정되지 않게 중단되었다가 다시 동작했을 때, 올바른 시점에서 동작을 재개하는가? 특히, 취소된 트랜잭션을 다시 재개하려고 시도하지는 않는가?
         - 오류의 조합이나, 반복되는 기능 오류가 시스템 충돌을 일으키지는 않는가?
         - 시스템 레벨에서의 side effect를 발생시키는 트랜잭션이 있는가?
         - 시스템을 중단하지 않고도 패치 또는 업데이트가 가능한가?
