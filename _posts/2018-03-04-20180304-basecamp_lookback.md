## 기술교육을 마치며
9주간 진행되는 베이스캠프도 마지막 Sprint만을 앞두고 있습니다. 프로젝트를 진행하고, 교육을 들으며 배웠던 점과 어려웠던 점은 어떤 것이 있었는지 정리해 보았습니다.

## 과제들
1. 프로젝트 기획서 작성
    - 큰 주제는 그룹웨어 중 이메일 서비스를 개발하는 것이었고, 요구사항에 따라 TF별로 기획서를 작성했습니다.
    - 학교 프로젝트 하던 것처럼 생각나는 기능은 일단 추가하고 보는 게 아니라, 기본 요구사항에 충실하면서도 '이걸 기간 안에 할 수 있을까?' 를 고민했습니다. 그러다 보니 메일 서비스 본래의 기능에만 집중할 수 있었습니다.
    - 수 차례 기획서를 썼다 지웠다 한 것 같습니다(ㅠㅠ)

2. Git/GitHub 교육
    - 브랜치를 이렇게 본격적으로 분리해보기는 처음이었습니다
        - git flow 짱짱
    - 프로젝트 진행 간 브랜치 잘 분리해서 개발은 했지만, 앞으로 커밋 메시지는 이슈번호나 두레이 업무번호를 붙이는 등 왜 그 커밋을 했는지에 대한 이유를 명확히 해야 할 것 같습니다.
3. HTTP 교육

4. HTML/CSS 교육

5. vi tutorial 교육
    - 명령어 목록 옆에 붙여놓고 시간 날때마다 외워야 될 것 같습니다. 정말 자주 쓰는 것 말고는 잘 외워지지 않아요=_=
    - [vim adventure](https://vim-adventures.com/) 는 정말 참신했습니다. 게임으로 편집기를 배우다니 :)

6. 스프링 프로젝트 생성하기 교육

7. JavaScript 교육

8. 데이터 모델링
    - 프로젝트를 설계하면서 가장 고민했던 부분이 아니었나 싶습니다.
    - 특히 '어느 정도까지 정규화를 해야하나?' 에 대해 TF 내에서 많은 고민이 오갔고, 장단점이 뚜렷했기에 쉽게 결정하지 못했습니다. 요구사항과 기획서까지 검토하며 모델링했던 것 같습니다.
    - Kafka같은 메시징 큐는 사용하지 않기로 했기에, RDB를 큐처럼 사용하는 신기한(...) 구조도 생각했었습니다.

9. 개발서버 설정 및 배포하기 교육
    - Apache -> Tomcat 순서로 종료하고, Tomcat -> Apache 순서로 시작합니다.
    - Tomcat을 먼저 종료했을 때, 잠깐이라도 보일 수 있는 5xx 오류를 피하기 위함입니다.

10. bash tutorial 교육
    - vi와 마찬가지로 연습이 많이 필요한 것 같습니다.

11. 프로젝트 개발
    - 이렇게 써놓으니 교육만 있을 것 같았지만 개발은 항상 진행하고 있었습니다 :D
    - 발표도 했구요.

12. 읽기 좋은 코드 교육
    - 그렇게 발표한 프로젝트가 내부적으로 어떤 이상한 코드를 지니고 있는지 알아보는 교육이었습니다.
    - 코드 컨벤션이 왜 중요한지를 배우고, 팀으로 일할 때 왜 남이 읽기 좋은 코드를 써야 하는지를 배웠습니다.
        - [오라클 자바 컨벤션](http://www.oracle.com/technetwork/java/codeconvtoc-136057.html)
        - 헷갈릴땐 꼭 읽읍시다.

13. CLI 과제
    - 웹 서비스 프로젝트의 MVC에서 Controller는 HTTP 요청에 담겨 온 변수들을 Service가 이해할 수 있는 변수로 변환해주는 역할 뿐입니다.
    - 즉, CLI 환경이더라도 기존 구현된 Service의 호출만으로 기능이 동작해야 합니다.
        - Service에는 웹에 의존적인(HTTP 등) 로직이 들어가지 않아야 합니다 :)

14. TDD with Junit4 교육
    - Jenkins CI를 이용해 테스트 커버리지는 항상 70% 이상으로 유지했습니다(line 수 기준)
    - Jenkins 보면서 이슈 줄여나가고, 테스트 커버리지 높이는 건 생각보다 재밌었습니다(ㅋㅋ)

15. 데이터 모델링 검수 결과 교육
    - FK는 실제 DB에 적용이 안 되더라도, 논리적으로 두 테이블이 관계를 갖는다면 ERD에 반영합니다.

16. 웹 서버 스케일 아웃 교육
    - 웹 서버가 두 대가 된다면, 세션과 같이 공통적으로 관리되어야 할 부분이 있습니다.
    - 세션은 Redis나 RDB를 이용해 관리할 수 있도록 했습니다. Redis는 정말 빨라요 :D

17. Object Storage 적용 과제
    - TC에도 S3같은 Object Storage가 있습니다!

18. OAuth 교육

19. DB서버 스케일 아웃 교육
    - 셀 아키텍처와 샤딩에 대해 배웠습니다.
    - 여기서부터 TF간 메일 전송이 가능하도록 인터페이스를 통일하게 됩니다(!!)

20. 안정적인 서비스 운영 교육
    - 죽지 않는 서비스를 개발하기 위해 고려해야 할 사항들을 배웠습니다.
    - 그간 운영서버에 배포하면서 오류가 발생하면 핫픽스 push하는것만 생각했었는데, 감당할 수 없는 장애에 대비해서 빠르게 롤백할 준비가 되어있는 게 더 중요한 것 같습니다=_=

22. QA
    - 남의 버그 찾는 건 생각보다 재밌었습니다.
    - 마찬가지로 저희 TF것도 남이 봐줬을 때 버그나 개선사항이 더 많이 보입니다(...)

21. 운영 스크립트, TC Log&Crash, 서비스 자동 재구동, nSight, Watchdog 적용 과제
    - 특정 시점의 로그를 확인하는 스크립트를 작성하고,
    - 에러 로그만 따로 관리하고
    - Tomcat 죽으면 살리고
    - 장비에서 메모리나 CPU를 과도하게 사용하거나, 로그인 등 특정 시나리오를 실패한다면 이메일, 메신저로 알림을 언제든지 뿜뿜 줄수 있게 되었습니다. 바로바로 대처할 수 있도록 합시다.

22. 성능 테스트 교육
    - 성능을 이야기할 때는 <b>응답 속도, 처리량</b>을 파악하고
    - <b>kill -3</b>으로 스레드 덤프는 꼭 남깁시다.


## 후기
- 중간중간 TF간, 멘토, 운영진 분들과도 코드리뷰도 진행했습니다. 혼자서 개발할 때는 발견하지 못했을 문제점을 여러 사람이 검토하여 훨씬 빨리 발견할 수 있었고, 읽기 좋은 코드에 대해 다같이 고민하여 코드가 더욱 견고해지는 느낌을 받았습니다.
- 학부 때 배웠던 지식들이, 실제 운영되는 서비스에서는 어떻게 적용되는지를 배우고 고민하는 동안 많이 확장된 것 같습니다. 서버는 어떻게 확장하는지, 어떻게 장애에 대비하고 대처하는지와 같은, 실제 운영을 해 보지 않는다면 몰랐을 내용들을 배울 수 있어서 너무 좋았습니다 :)
- 잘 동작하는 코드는 중요하지만, 그 코드를 남이 이해할 수 있도록 만드는 것과, 테스트를 통해 맘 편히(!) 코드를 변경할 수 있도록 만드는 것이 더 중요하다는 것을 배웠습니다.
- 다 적진 못했지만 정말 많은 내용이 9주라는 기간 안에 녹아 있었습니다. 기획부터 개발, 리팩토링과 유지보수는 물론이고, 개발에 필요한 사내 인프라를 신청하고 이용하는 것, 주어진 시간 안에서 팀의 일정을 관리하는 것들을 직접 체험하며 배웠습니다. 기간 안에 많은 것들을 체계적으로 배울 수 있었던, 실제 회사에서의 개발을 알려주는 좋은 과정이었다고 생각합니다.
- 기술교육 기간 동안 많은 것을 알려주신 멘토님, 교육자, 운영진 분들께 감사드립니다 :D

- 마지막 Sprint에도 교육이 빡빡하게 잡혀 있습니다. 오예
