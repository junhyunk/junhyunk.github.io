### Part 3 를 읽고 테스트 환경에 대해 정리하시오. (Word 1 page 분량)

- 성능 테스트 효율성을 증가시키기 위한 시스템 평가
    - 개요
        - 시스템 평가는 성능 테스트를 수행하는 동안 병행되는 프로세스이지만, 초기 테스트 프로젝트에서 수행 더 큰 가치를 제공함.
        - 시스템 평가의 관심사는 성능 테스트가 프로젝트의 특정 요구사항을 만족시키는 데 도움을 주는 프로젝트 전반에 관한 정보(시스템의 기능, 예상되는 사용자의 행동, 시스템 아키텍처 등등)를 수집하는 것임.
    - 시스템 평가를 위한 접근 방법
        - 시스템을 평가하는 것은 다음과 같은 사항을 포함하지만, 제한하는 것은 아님.
            1. 시스템의 사용자 지향 기능 식별
            2. 사용자가 시작하지 않은 프로세스 및 기능 식별
            3. 예상되는 사용자의 활동 확인
            4. 예상되는 것 이상의 잠재적 사용자의 활동 수립
            5. 테스트 및 프로덕션 아키텍처의 정확한 모델 개발
            6. 실제 사용자 환경에서의 합리적인 모델을 개발
            7. 해당 아키텍처를 사용하는 다른 프로세스/시스템을 확인
        
    - 이러한 활동들은 다음과 같은 단계로 수행될 수 있음.
    1. 시스템의 기능이나 비즈니스 프로세스를 capture
        - 성능 허용 기준을 수립하기 위한 시스템의 핵심 기능을 식별함.
        - 그 뒤, 작업 모델은 성능 허용 기준 및 시스템 기능들을 검증하도록 평가될 수 있음.
        - 성능 테스트에서는, 테스트 중인 시스템의 핵심 기능을 식별하는 것은 매우 중요함. 이 작업은 성능 허용 기준을 초기에 결정하는 것과, 사용자 커뮤니티 모델을 수립하는 데 도움을 줌.
        - 이 단계를 수행하기 위해서는 시스템의 모든 기능을 이해해야 하고, 이해 관계자와 함께 시작하는 것이 좋음.
        - 시스템 기능을 결정하는 데 좋은 자원들은 다음을 포함함.
            1. 이해 관계자들과의 인터뷰
            2. 계약
            3. 유사한 어플리케이션들에 대한 사용 정보
            4. 고객의 기대사항
            5. 유사한 어플리케이션들에 대한 경험
            6. 설계 문서
            7. 상태천이도
            8. 요구사항 및 유스케이스
            9. 마케팅 자료
            10. 프로젝트 계획
            11. 비즈니스 주기
            12. 핵심 비즈니스 프로세스

    2. 사용자의 활동을 capture
        - 테스트 중인 어플리케이션을 위한 핵심적인 사용자 활동을 식별함.
        - 성능 테스트에서 모든 가능한 사용자 태스크나 활동을 시뮬레이션하기는 비현실적이기 때문에, 어떤 활동이 가장 중요한지 정할 필요가 있음.
            - 가능한 사용자 활동이 무엇인지 먼저 정해야 함.
        - 사용자의 활동(오타인듯)을 결정하는 데 좋은 자원들은 다음을 포함함.
            1. 유사한 어플리케이션들에 대한 사용 정보
            2. 고객의 기대사항
            3. 유사한 어플리케이션들에 대한 경험
            4. 요구사항 및 유스케이스
            5. 이해 관계자들과의 인터뷰
            6. 마케팅 자료
            7. 도움말 및 설명서
            8. 고객 조직도(?)
            9. 네트워크 또는 보안 matrix
            10. 과거 데이터(청구서, 웹 로그 등등)
            11. 주요 비즈니스 주기

    3. 논리적, 물리적 아키텍처를 capture
        - 어플리케이션과 하드웨어, 소프트웨어의 구조 관계를 식별
        - 특정한 문제를 다루는 성능 테스트를 설계하는 것과, 성능 병목점을 찾는 데 중요함.
        - 시스템 아키텍처에 대한 이해가 부족하다면, 성능 테스트를 수행하는 데 부정적인 영향을 끼칠 수 있고 차후 튜닝 프로세스에서 시간이 추가적으로 소요될 수 있음.
        
        - 논리적 아키텍처
            - 논리적 아키텍처란 소프트웨어나 코드의 구성, 상호작용, 추상화를 의미함.
            - 코드는 전체 어플리케이션에서 객체, 함수, 클래스를 포함할 수 있음.
            
            1. 클라이언트 계층
                - 요청된 데이터를 표시함.
            2. 프리젠테이션 계층
                - 모든 비즈니스 로직을 관리하고, 클라이언트에게 데이터를 제공함.
            3. 데이터 저장소 계층
                - 시스템에 의해 사용되는 데이터를 유지함(보통 DB).

        - 물리적 아키텍처
            - 소프트웨어가 구동되는 실제 하드웨어의 물리적 아키텍처는 명확해야 함(논리적 아키텍처와 마찬가지로 중요).
            - 실제 하드웨어를 '환경' 또는 '네트워크 아키텍처'라고 많이들 언급하지만, 두 용어 모두 성능 테스터의 관심사를 모두 포함하지는 않음.
        
        - 시스템 아키텍처
            - 시스템 아키텍처는 논리적, 물리적 아키텍처를 통합하여 정리한 것임.
