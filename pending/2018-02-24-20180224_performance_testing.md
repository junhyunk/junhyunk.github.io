- 테스트의 목적
    - 어플리케이션에 대한 전반적인 테스트를 통해 이해관계자들의 의사결정에 도움을 줌.
    - 추가적으로, 성능 테스트는 시스템의 병목점을 식별하거나, 튜닝, 향후 테스팅을 위한 사전작업, 그리고 성능 목표와 요구사항을 결정하는 데 사용되는 경향이 있음. 또한, 성능 테스트 결과를 통해 배포 시 필요한 하드웨어 구성을 가늠할 수 있음.


### Chapter 1 을 읽고 성능 테스트의 정의를 정리하시오. (Word 1 page 분량)
- 성능 테스트는 반응성(responsiveness), 처리량, 신뢰성을 결정하거나 주어진 작업량에 대한 시스템의 확장성을 판단하는 데 목적을 둠.
- 성능 테스트는 다음과 같은 것을 이루기 위해 수행됨
    - 제품의 준비도
    - 성능 기준에 대해 실행(?)
    - 다중 시스템이나 설정에 대한 성능 특성을 비교
    - 성능 문제 식별
    - 시스템 튜닝
    - 처리량 식별

- 성능 테스트 핵심 activity
    1. 테스트 환경 식별하기
        - HW, SW, 네트워크 설정 등등
    2. 성능 기준 식별하기
        - response time, throuput, resource utilization goals, constraint 등등
    3. 테스트 계획 및 설계하기
        - 시나리오, 대상 사용자, 테스트 데이터, 수집할 데이터 등등 정의
    4. 테스트 환경 구성하기
        - 테스트 환경, 데이터, 결과값 검증
        - 검증된 테스트 수행해야 됨
    5. 테스트 설계 구현하기
    6. 테스트 실행
    7. 결과 분석, 보고, 테스트 재실행하기
        - 결과 공유..

- Project Context(이하 '프로젝트 맥락')
    - 성능 테스트가 성공적으로 수행되기 위해서는, 성능을 테스트하는 것과 그 자체를 테스트하는 것이(?) 프로젝트 맥락과 관련이 있어야 함(쓸데없는 테스트 하지 말란 소리인 듯)
    - 프로젝트 맥락에 대한 이해가 없다면, 정작 중요한 건 테스트 못해서 시간낭비에, 좌절에, 충돌 등등 안좋음.
    - 프로젝트 맥락은 다음과 같음(아래 항목으로 제한하는 건 아님)
        - 프로젝트의 전체적인 비전(목적)과 의도
        - 성능 테스트의 목적
        - 성능 기준
        - 개발 라이프사이클
        - 프로젝트 일정
        - 프로젝트 예산
        - 가용한 도구와 환경
        - 성능 테스트를 수행하는 팀의 스킬셋
        - 파악된 성능 이슈에서의 우선순위
        - 그지같이 동작하는 어플리케이션을 배포했을 경우의 business impact

- 프로젝트의 비전
    - 프로젝트의 비전은 어떤 성능 테스트가 중요하고 가치가 있을지 결정함.
    - 비전이 변경될 가능성이 있다면, 가능한 자주 확인할 것.
- 시스템의 목적
    - 테스트할 어플리케이션이나 시스템의 목적을 이해할 것.
    - 테스트에서 중점을 두어야 하는, 가장 우선순위가 높은 성능 특성을 식별하는 데 도움을 줌.
    - 시스템의 관심
